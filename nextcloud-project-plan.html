Architecture

VPC / IGW │ ├── EC2 (from Packer AMI) ── IAM instance role │ │ (server_ip output; user-data runs Phase 7 on first boot) │ ├── RDS MySQL (app database) │ └── S3 bucket (primary file storage + branding asset) │ └── Elastic IP ── DNS/TLS (deferred — no domain yet) infra/bootstrap/ (state) ── onevoice-storage/terraform/ (flat) + onevoice-storage/packer/ terraform apply + Packer rebuild — run manually/locally

Actual layout diverged from the original phase-split plan: the main stack ended up flat under onevoice-storage/terraform/ (vpc.tf, iam.tf, db.tf, s3.tf, compute.tf, data.tf) instead of a networking.tf/iam.tf/database.tf/compute.tf split inside infra/main/. infra/bootstrap/ stayed where it was; the main stack reads it via data "terraform_remote_state". Packer moved to onevoice-storage/packer/. State backend is S3 only, SSE-KMS, versioning on, use_lockfile = true for native S3 locking — no DynamoDB lock table. No CI/CD pipeline — Jenkins was descoped in favor of self-managed applies.

Known Issues & Fixes Log

Consolidated across Phases 4, 5, 7, and Dropbox-migration prep work. Ordered roughly by where each was hit.

Packer component variable
Resolved

Issue: uncertainty whether component = "clixx" was a leftover from a prior template or intentional.

Fix: confirmed intentional — default value is "nextcloud".

Terraform templatefile() variable collision
Resolved

Issue: user-data.sh uses bash-native ${...} syntax (especially ${!NEW_USERS[@]} array expansion), which templatefile() parses as HCL interpolation — breaking the script.

Fix: escaped all bash-only variables as $${...}. Only the 13 real template variables (db_password_ssm_path, admin_password_ssm_path, db_host, db_name, db_user, admin_user, s3_bucket, aws_region, elastic_ip, public_dns, organization, environment, domain_name) keep single ${...}.

Amazon Linux 2023 web server user
Resolved

Issue: occ commands failed under the Debian/Ubuntu assumption (www-data) — AL2023 doesn't use that user.

Fix: confirmed correct user is nginx (via ps aux); all occ commands run as sudo -u nginx php occ ....

Deprecated theming parameter
Resolved

Issue: occ theming:config didn't apply correctly using the color parameter.

Fix: use primary_color instead — color is deprecated.

Theming CSS caching
Open / Deferred

Issue: logo/branding changes didn't appear immediately after occ theming:config ran, due to server-side CSS caching.

Fix: not fully resolved — custom CSS for logo sizing also deferred. Likely needs a cache-clear step added to the theming workflow.

S3 primary objectstore vs. human-readable paths
Resolved

Issue: assumed migration files could be uploaded directly into the primary Nextcloud S3 bucket. Doesn't work — primary objectstore uses internal keys (urn:oid:xxxx), not real paths, so direct writes are invisible to Nextcloud.

Fix: used a separate dedicated migration S3 bucket, mounted as External Storage, indexed with occ files:scan --all. Migration executed and complete.

CloudWatch agent config file path assumption
Resolved

Issue: assumed the agent's editable config lived at /opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json — that path doesn't exist on this instance.

Fix: the real file-sourced config is /opt/aws/amazon-cloudwatch-agent/etc/config.json (mirrored to amazon-cloudwatch-agent.d/file_config.json, translated to .toml at runtime). Confirmed via ls on the instance and cross-checked against live CWAgent-namespace metrics already flowing. No SSM parameter involved — plain local file, edited directly and re-applied via amazon-cloudwatch-agent-ctl -a fetch-config -m ec2 -s -c file:....

AWS CLI / s5cmd profile ambiguity
Resolved

Issue: multiple local AWS profiles risked uploading migration data to the wrong account/bucket.

Fix: explicitly set $env:AWS_PROFILE = "onevoice" or s5cmd --profile onevoice; verify target account first with aws sts get-caller-identity --profile onevoice.

EBS data volume AZ mismatch
Resolved

Issue: compute.tf had the EBS data volume's AZ hardcoded to us-west-1a, mismatched against the us-east-1 stack.

Fix: changed to us-east-1a and committed.

S3 branding-object bucket reference
Resolved

Issue: s3.tf's branding upload resource referenced bucket_domain_name instead of bucket — wrong attribute for the logo object.

Fix: changed to reference bucket and committed.

DB hardening settings loosened
Open Decision

Issue: not a bug, but a flagged risk — db.tf's skip_final_snapshot / deletion_protection changed to true / false (from false / true) to speed up iteration.

Fix: none applied — needs an explicit decision on flipping back before treating the DB stack as production-final.

Groupfolders app not visible in UI
Guidance given

Issue: app not found browsing Settings → Apps.

Fix: lives under the "Organization" category filter in the left sidebar, not Featured/Productivity. If missing there too, the instance likely can't reach apps.nextcloud.com to populate the catalog — check Settings → Administration → Overview for a connectivity banner, or run occ app:install groupfolders via CLI to surface the real error.

UI-installed apps don't survive instance replacement
Fix identified — not implemented

Issue: groups, folder assignments, and files live in RDS/S3 and would survive instance termination + AMI-relaunch — but groupfolders was enabled manually via the UI, never baked into the AMI or scripted into user-data.sh. A future replacement instance would come up without it installed.

Fix (recommended): add occ app:install groupfolders + occ app:enable groupfolders to user-data.sh, matching the pattern used for the rest of Phase 7.
